# 异步求助包竞品调研

用途：查清楚市场上是否已有“把问题现场打包给别人处理”的成熟产品，并提炼可迁移机制。

## 1. 最接近的产品类别

这些竞品原本不是给家庭使用的。它们大多服务产品团队、客服团队、测试团队：当客户或测试人员发现软件问题时，不靠文字长篇解释，而是把屏幕录制、截图、操作路径和技术信息打包给工程师，让工程师更快复现和修 bug。

我们借鉴的是它们的“问题打包机制”，不是它们的用户对象和商业场景。

### 1.1 录屏链接/客户问题上报

代表：Jam Recording Links、Loom、ScreenRec、ScreenPal。

机制：支持方发一个链接，用户打开后录屏，系统生成可分享链接；部分产品会自动生成标题、摘要、复现步骤。

可借鉴：不用实时开会；不用长篇文字描述；视觉材料直接给处理者。

不足：用户仍要理解“录屏”这件事；产品语境是团队协作/客户支持，不是妈妈向女儿求助。

### 1.2 网页视觉反馈/bug 上报

代表：Marker.io、BugHerd、Usersnap、Userback、BetterBugs。

机制：用户点击网页上的反馈按钮，提交截图、标注、屏幕录制；系统自动带上 URL、浏览器、操作系统、控制台日志、网络请求、最近几十秒 session replay。

可借鉴：一键报告、自动上下文、减少支持方反复追问、状态回传。

不足：通常只覆盖自己接入了 SDK 或脚本的网站，不能直接跨手机 App 使用。

### 1.3 移动 App session replay / crash report

代表：Gleap、UXCam、Sentry Mobile Replay、Bugsee、Datadog Mobile Session Replay。

机制：在 App 内集成 SDK，自动记录用户点击、滑动、页面变化、错误、网络请求，开发者可回看问题发生前后的过程。

可借鉴：自动捕捉复现过程、隐私遮罩、只在授权后上传、把“用户说不清”转化为“支持方可回看”。

不足：只能用于开发者自己的 App；不适合直接捕捉妈妈在抖音、微信、地图、医院公众号里的跨 App 问题。

## 2. 共同机制

- 触发要低摩擦：按钮、链接、widget、摇一摇、录制请求。
- 问题要视觉化：截图、录屏、session replay，而不是纯文字。
- 上下文要自动化：页面、设备、时间、操作步骤、环境信息自动带上。
- 支持方要少追问：报告里尽量包含“复现路径”。
- 结果要可流转：链接、工单、状态更新、转交给其他人。
- 隐私要可控：敏感区域遮罩、权限确认、只录必要内容。

## 3. 对本项目的可迁移设计

把 B2B bug report 翻译成“妈妈求助包”：

| 竞品机制 | 妈妈版转译 |
| --- | --- |
| Report a bug 按钮 | “我需要帮忙”按钮或语音唤起 |
| 录屏/截图 | 产品语音提示妈妈复现刚刚的问题并自动录制 |
| 技术日志 | App 名、页面文字、妈妈语音、关键截图、发生时间 |
| AI 生成复现步骤 | AI 给女儿整理“妈妈刚刚做了什么、卡在哪里” |
| 工单链接 | 发给女儿的求助链接 |
| 指派开发者 | 女儿查看；必要时转给可信朋友 |
| 状态流转 | 女儿回复“我看到了/稍后帮你/需要通话” |

## 4. 项目仍有差异化的地方

- 现有工具服务的是公司客户支持，不服务亲密关系里的“我怕打扰你”。
- 现有工具通常要求用户进入某个网页/产品内上报，无法覆盖妈妈跨 App 的真实手机问题。
- 现有工具关注工程复现，本项目关注女儿理解妈妈的处境并给出可执行反馈。
- 现有工具不处理“能否转交可信朋友”“妈妈是否感到被责备”等关系问题。

## 5. 对黑客松 MVP 的约束

MVP 不能只是展示稿，必须能被真实使用，哪怕只支持一个场景、流程很简陋。

第一版最低可用能力：

1. 妈妈端能提交截图或录屏。
2. 妈妈端能补充一段语音或文字。
3. 系统能生成一个求助链接。
4. 女儿端能打开链接查看材料。
5. AI 能根据截图/录屏和补充说明生成摘要与下一步建议。
6. 女儿能给妈妈一个可执行回复。

如果系统级自动录屏来不及做，就不要假装已经做成；第一版可以先引导妈妈使用系统录屏/截图，或直接从相册上传。

## 6. 对页面的启发

MVP 可以把求助包做成一个清晰页面：

- 顶部：妈妈的语音描述转文字。
- 中间：录屏/截图时间线。
- 右侧或下方：AI 给女儿的摘要、可能原因、下一步建议、风险提醒。
- 底部：女儿可选回复：“我看到了”“你先别点”“我稍后看”“我们通话一下”。
